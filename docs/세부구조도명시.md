# 🚀 Project Standard & Convention: AI Career Advisor

> 이 문서는 `ai_career_advisor` 프로젝트의 패키지 구조, 명명 규칙 및 개발 표준을 정의합니다. 모든 팀원은 본 가이드를 준수하여 코드 일관성을 유지합니다.

---

## 1. 프로젝트 기본 설정
* **Root Package:** `com.example.ai_career_advisor`
* **Java Version:** Java 17+
* **Build Tool:** Gradle (추천)

---

## 2. 패키지 표준 구조 (Standard Skeleton)
계층형 구조를 기반으로 하며, 각 역할에 따라 엄격히 분리합니다.

| 패키지명 | 설명 |
| :--- | :--- |
| **Config/** | 애플리케이션 설정 (Security, Swagger, DB 설정 등) |
| **Entity/** | JPA 엔티티 (DB 테이블 매핑) |
| **Repository/** | Spring Data JPA 인터페이스 |
| **DTO/** | 데이터 전송 객체 (Request/Response용) |
| **Service/** | 비즈니스 로직 인터페이스 |
| **ServiceImpl/** | 비즈니스 로직 구현체 |
| **Controller/** | API 엔드포인트 제어 (REST API) |
| **Constant/** | Enum 및 공통 상수 관리 |
| **Util/** | `@Component` 기반의 공통 유틸리티 클래스 |
| **Mapper/** | `@Mapper`, `@Mappings` (Entity ↔ DTO 변환) |

---

## 3. Entity 도메인 분리 (Domain-Driven)
가독성과 확장성을 위해 `Entity` 하위 패키지를 도메인 성격에 따라 세분화합니다.

* **`Entity.user`**: 사용자 기본 정보 및 계정 관련
* **`Entity.master`**: 시스템 공통 코드, 마스터 데이터
* **`Entity.profile`**: 서비스 핵심 데이터 영역
    * `..profile.core`: 프로필 루트, 목표(Goal), 스냅샷
    * `..profile.mapping`: N:M 관계 (Skill, Value, Env, Concern 등)
    * `..profile.experience`: 1:N 관계 (경험 이력 상세)

---

## 4. 명명 규칙 (Naming Convention)
DB 테이블과 Java 클래스 간의 추적성을 위해 **"토큰 일치(Token Matching)"** 전략을 적용합니다.

### [DB Table ↔ Entity Class]
* **DB 테이블**: `snake_case` (Prefix 활용 필수)
* **Entity 클래스**: 테이블명을 `PascalCase`로 변환하여 1:1 매핑

| 구분 | 예시 A (사용자) | 예시 B (프로필 스킬) |
| :--- | :--- | :--- |
| **DB Table 명** | `app_user` | `prf_profile_skill` |
| **Entity 클래스명** | `AppUser` | `PrfProfileSkill` |

## 5. 주석 및 문서화 규칙 (Documentation)
협업 시 기능의 "계약서" 역할을 하도록 상세히 기술합니다.

### 5.1 소스 코드 주석 (고정 규칙)
* **클래스 상단 (Javadoc)**: 역할, 도메인, 연관 테이블, 핵심 제약조건(Unique/Index) 명시.
* **필드 단위 주석**: 단순 타입 설명이 아닌 실제 **"업무적 의미"** 기술.
* **Service 메서드**: 처리 흐름(입력 → 검증 → 저장 → 예외 처리 → 반환) 명시.

### 5.2 문서 관리 (`/docs`)
* `db-schema.md`: 테이블 설계 및 제약 조건.
* `pipeline.md`: 프로필 분석부터 자소서 생성까지의 프로세스 흐름.
* `api-contract.md`: Request/Response JSON 규격 및 에러 코드 정의.

## 6. 단계별 진행 방식 (Modular Delivery)
답변의 정확도를 위해 아래 파트 순서대로 작업을 진행합니다.

1.  **Part A**: 컨벤션 및 폴더 구조 확정
2.  **Part B**: DDL 및 Entity 구현 (주석 포함)
3.  **Part C**: Repository, Service, Controller 구현 (**"반드시 변수로 받아서 return"** 규칙 적용)
4.  **Part D**: 화면 설계 및 파이프라인 문서화 (.md)
