# 1 ) 고용24(워크24) API를 “연관되게” 쓰는 구조

고용24 API들은 서로 독립적으로 보이지만, 실제로는 **명확한 연관 흐름**을 따라 사용하도록 설계되어 있습니다.  
핵심 체인은 다음과 같습니다.

**(심리검사결과) → (직업코드 추출) → (분류코드 매핑) → (직업/채용 API로 확장)**

---

## 📌 공식 문서에 명시된 연결고리

한국고용직업분류 매핑코드(214D02) API의 `jobsIcd` 파라미터는  
**심리검사결과 API 출력의 `<jobInfo1>`, `<jobInfo2>` 직업코드를 그대로 넣으라고 명시**되어 있습니다.  
→ 즉, **“심리검사결과 API 결과가 214D02의 입력값이 된다”**가 공식적인 연결 구조입니다.

---

## 📌 추천 연관 구조도(서비스 관점)

### 1) 심리검사결과 조회 (개인 식별키 기반)
- 대학/기관 연계 등 사전 조건 필요
- 결과에는 추천직업, 직업코드 등이 포함
- 출처: 데이터.go.kr

### 2) 직업코드 → 한국고용직업분류코드 매핑 (214D02)
- 심리검사 결과의 `jobInfo1`, `jobInfo2`를 `jobsIcd`로 전달
- 분류코드 리스트 획득
- 출처: 고용24

### 3) 직업정보 API로 “설명/콘텐츠” 강화
- 직업명, 직업코드, 분류정보 조회
- 직업 설명/특성/관련 정보 제공
- 출처: 데이터.go.kr

### 4) 채용정보 API로 “실제 채용공고” 연결
- 키워드, 직종코드, 지역 등으로 검색
- 워크넷 채용정보 기반
- 출처: 데이터.go.kr

---

## ⚠️ 실무에서 꼭 짚어야 할 포인트

### 1) 코드 체계가 100% 동일하다고 단정하면 위험
- 심리검사 결과의 직업코드/분류코드
- 채용정보 API의 `occupation` 직종코드  
  → 서로 **완전히 동일한 코드 체계가 아닐 수 있음**

### 2) 내부 매핑 테이블(DB) 필요
- 공공데이터포털에도 직종코드 데이터셋이 별도로 존재
- 서비스에서는 보통 **Canonical Code(표준 코드 계층)** 을 두고  
  모든 API의 코드를 여기에 매핑하는 방식이 안정적

---

## 📌 전체 흐름 요약
> [심리검사 결과 API] ↓ jobInfo1/jobInfo2 [214D02 직업분류 매핑 API] ↓ 분류코드 [직업정보 API] ↓ 직업 상세정보 [채용정보 API] ↓ 실제 채용공고 [사용자에게 추천]
